# Hourly Demand & Restaurant Performance Analytics — Power BI Dashboard

An end-to-end Power BI analytics project that analyzes food-delivery order volume by time-of-day, restaurant, city, and account manager territory, built to help operations teams staff kitchens/riders and manage restaurant-partner relationships.

> **Note on data:** This is a portfolio recreation of a real dashboard I built for my employer. All company names, restaurant names, staff names, and order records in this repository are **synthetically generated** and do not represent any real business, person, or transaction. The original production version (with real data) is confidential and is not published here. See [Data Disclaimer](#data-disclaimer) below.

---

## Project Overview

Food-delivery platforms live and die by two things: getting orders to the right kitchens at the right time, and managing hundreds (or thousands) of restaurant partners profitably. This dashboard was built to answer a recurring operational question:

> *"When exactly are we busiest, which restaurants/cities drive that demand, and which account manager territories need attention?"*

It ingests raw order-level transaction exports and a restaurant partner master list, transforms them in Power Query, and presents the result as an interactive, slicer-driven Power BI report across six analytical pages.

## Business Problem

Operations and account-management teams needed a single source of truth to:
- Understand demand concentration at the **hour / 30-minute / 15-minute** level to plan kitchen and rider capacity
- Compare order volume **day-by-day within a month** at the restaurant level
- See performance broken down by **city** to support location-level operational decisions
- Give each **account manager** a live view of the order volume generated by the restaurants in their portfolio
- Investigate a recurring **operational pattern on Sundays** versus the rest of the week

## Objectives

1. Build a repeatable Power Query pipeline that ingests order-history export files and a restaurant master file, and combines them into a clean analytical model.
2. Model the data as a star-schema-style structure (fact table of orders + restaurant dimension) connected on a single relationship key.
3. Engineer time-bucket columns (hourly, 30-minute, 15-minute) directly in the model so the same fact table can support three different granularities without duplicating data.
4. Build slicer-driven pages so non-technical stakeholders (ops managers, account managers) can self-serve answers without needing to write queries.
5. Surface a Sunday-by-Sunday comparison — flagging each Sunday in the data with its occurrence number within the month — to support a real recurring business question about demand shifts at the start of the week.

## Data Model

**Tables**

- **sample_orders (fact table)** — one row per order, 25,000 sample rows (synthetic). Grain: individual order transaction.
- **sample_restaurant_master (dimension table)** — one row per restaurant branch, 450 sample rows (synthetic). Contains chain/restaurant name, city, cuisine, account manager, commission %, and status.
- **Date tables** (auto-generated by Power BI, one per date-typed field used in slicers)

**Relationship**

```
sample_orders[Restaurant Branch ID]  *  ──────  1  sample_restaurant_master[restaurant_branch_id]
              (many)                                          (one)
              Cardinality: Many-to-One | Cross-filter: Single direction | Active
```

This is intentionally a simple two-table star schema — the fact table carries all transactional detail, and the dimension table carries the descriptive/organizational attributes (chain, city, account manager) that the report slices by.

## Power Query (M) Transformations

**sample_orders table**
- Loaded from a CSV export
- Explicit type conversions applied to all columns (dates, currency/decimal amounts, whole numbers, booleans)
- Two derived columns added at the query stage: `Month Name` (`Date.MonthName`) and `Date` (`DateTime.Date`, stripping the time component for date-level slicing)

**sample_restaurant_master table**
- Loaded from a CSV export
- Headers promoted from the first data row
- Explicit type conversions applied to all columns

*(Full annotated M code and background on the original folder-based ingestion pattern is in [`docs/power_query_transformations.md`](docs/power_query_transformations.md).)*

## DAX Measures & Calculated Columns

**Explicit measures** (aggregations reused across the report):

| Measure | Logic |
|---|---|
| `Total Orders` | `COUNTROWS('sample_orders')` |
| `Total Revenue` | `SUM('sample_orders'[Total Amount])` |
| `Avg Order Value` | `DIVIDE([Total Revenue], [Total Orders])` |
| `Cancellation Rate %` | Share of orders with status `cancelled`, `restaurant_rejected`, or `not_delivered` |

**Calculated columns** (computed once at the model layer so every visual reuses the same buckets):

| Column | Table | Logic |
|---|---|---|
| `1 Hour Range` | sample_orders | Formats the order's hour into a `"HH-HH"` band, e.g. `13-14` |
| `30 Min Range` | sample_orders | Buckets the order into a 30-minute window, e.g. `13:30-14:00` |
| `15 Min Range` | sample_orders | Buckets the order into a 15-minute window, e.g. `13:15-13:30` |
| `1 Hour Sort` / `30 Min Sort` / `15 Min Sort` | sample_orders | Numeric helper columns (minutes-since-midnight based) so the text-formatted range columns sort chronologically instead of alphabetically — wired up via Power BI's "Sort by Column" |
| `Sunday Occurrence` | sample_orders | Checks if the date is a Sunday (`WEEKDAY(d,1) = 1`), and if so labels it with its occurrence number within the month via `ROUNDUP(DAY(d)/7, 0)` — e.g. the first Sunday of the month becomes `"Sunday 1"`, the second `"Sunday 2"`, and so on. Non-Sundays return blank. This powers the Sunday Comparison page, letting every Sunday across the whole dataset be compared side by side rather than only the two most recent |

*(Full DAX with explanatory comments is in [`docs/dax_measures.md`](docs/dax_measures.md).)*

## Interactive Features

- **Cross-filtering slicers** on Month, Date, and Hour Range, present on the relevant pages
- **Pivot (matrix) tables** with drill-down and hierarchical date columns (day within month)
- **Clustered column charts** for at-a-glance demand-by-time-bucket visuals on the Overview page
- **KPI cards** (Total Orders, Total Revenue, Avg Order Value, Cancellation Rate %) on the Overview page
- Every visual respects **page-level and visual-level filters**, so an account manager can isolate just their own portfolio

## Key Insights *(pattern shown on synthetic data — replace with your own real findings before presenting)*

- Demand is concentrated in two clear peaks — a smaller midday peak and a larger evening peak — visible immediately from the 15-minute and hourly bucket charts
- A small number of restaurants account for a disproportionate share of total order volume, visible in the Restaurant Wise pivot
- Order volume is not evenly distributed across cities, which has direct implications for rider allocation by location
- The Sunday Comparison page makes it possible to see whether order volume trends up, down, or stays flat across successive Sundays within a month, which can inform staffing decisions ahead of time

## Data Disclaimer

The `/data` folder in this repository contains **fully synthetic data**, generated with a Python script (`data_generator_script.py`, included in this repo) to match the *shape and statistical distribution* of the original dataset (order volumes, hour-of-day patterns, amount distributions) — **no real restaurant, customer, rider, or employee data is included anywhere in this repository.** Column names such as `Foodi Burn` in the original file (a brand-specific subsidy metric) have been renamed to generic equivalents (`Platform Subsidy`). Real employee/account-manager names, real restaurant chain names, real customer phone numbers, and real file paths from the original workbook have all been replaced or removed. See [`docs/anonymization_checklist.md`](docs/anonymization_checklist.md) for the full list of what was changed and why.
